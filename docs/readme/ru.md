## О проекте

Данный шаблон реализует облегчённую архитектуру для CRUD-приложений и микросервисов.

Архитектура создана как компромисс между двумя крайностями, которые часто встречаются в реальных проектах.

С одной стороны находятся простые CRUD-приложения, в которых бизнес-логика быстро начинает зависеть от конкретных технологий:

* FastAPI
* SQLAlchemy
* PostgreSQL
* Redis
* Kafka
* внешних API

По мере роста проекта это приводит к сильной связанности кода и усложняет замену инфраструктурных компонентов.

С другой стороны находится полноценный Domain-Driven Design (DDD), который вводит большое количество дополнительных концепций:

* Доменные сущности (Entities)
* Агрегаты (Aggregates)
* Объекты-значения (Value Objects)
* Доменные события (Domain Events)
* Use Cases
* Application Services

Для большинства CRUD-сервисов такой уровень абстракции оказывается избыточным и приводит к усложнению разработки без существенной практической выгоды.

### Цель архитектуры

Основная цель архитектуры — сохранить простоту CRUD-приложения, не теряя преимуществ разделения ответственности между слоями.

Архитектура позволяет:

* заменять базы данных без изменения бизнес-логики;
* заменять внешние интеграции и брокеры сообщений;
* заменять транспортный слой (REST, gRPC, GraphQL);
* изолировать бизнес-логику от инфраструктуры;
* сохранять простые и понятные точки входа в систему;
* избегать избыточных абстракций.

### Основные концепции

Архитектура строится вокруг небольшого набора компонентов:

* API Layer
* DTO
* Services
* Ports
* Infrastructure

### Философия архитектуры

Бизнес-логика должна быть независима от используемых технологий.

Сервисы не должны знать:

* какой используется веб-фреймворк;
* какая используется база данных;
* каким образом реализована авторизация;
* через какие библиотеки выполняются интеграции.

Вместо этого сервисы работают через контракты (Ports), а инфраструктурный слой предоставляет конкретные реализации.

```text
API
 ↓
DTO
 ↓
Service
 ↓
Port
 ↑
Infrastructure
```

Такой подход позволяет получить чистое и независимое ядро приложения, сохраняя простоту разработки и возможность дальнейшего развития проекта без перехода к полноценной DDD-архитектуре.
